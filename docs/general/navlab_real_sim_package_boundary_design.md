# NavLab Real / Sim Package Boundary Design

## 1. Goal

`navlab` should be the runtime package. `orchestration` should be the control
plane.

The current tree mixes these responsibilities:

- `navlab` contains ROS/MAVLink runtime nodes.
- `orchestration/src/tasks` contains CLI task wrappers, but also contains some
  runtime nodes and direct MAVLink task behavior.

This design defines the target package boundary before moving code.

## 2. Target Boundary

Top-level runtime code is split by safety domain first:

```text
navlab/
  common/
  real/
  sim/
```

The safety-domain boundary is more important than the historical package names.
The target steady state is that `navlab` has no other top-level runtime packages.
Former top-level packages such as `navlab.companion`, `navlab.gazebo_sensor`,
`navlab.slam`, and `navlab.interfaces` are collapsed into `common`, `real`,
or `sim`.

- `navlab.real.*`: may touch real hardware, real serial ports, real FCU, motors,
  or real sensors.
- `navlab.sim.*`: may touch Gazebo, SITL, simulation sensors, replay-only data,
  or simulation-only helpers.
- `navlab.common.*`: only pure utilities that are identical in real and sim.

`orchestration` remains outside this runtime tree:

```text
orchestration/
  src/
    cli.py
    configs/
      project_config.py
      run_config.py
      task_config.py
    runtime/
    tasks/
```

`orchestration` owns process startup, config loading, CLI UX, doctor chains,
summary artifacts, and task ordering. It should not own runtime ROS nodes or
direct vehicle behavior.

Language-neutral runtime contracts live beside both packages:

```text
contracts/
  proto/
    navlab/
      runtime/
        v1/
          task_result.proto
```

`contracts` is not runtime code and not orchestration code. It defines the file
artifact schema that both sides agree on.

## 4. Placement Rules

Use these rules when adding or moving files:

| Code kind | Target package |
| --- | --- |
| Logging setup, TOML parsing, process-manager helpers, pure JSON/status codecs | `navlab.common` |
| Direct MAVLink arm/disarm, EKF source switching, FCU status for real arming | `navlab.real.common` or `navlab.real.companion.nodes` |
| Real serial lidar/rangefinder bridges | `navlab.real.companion.nodes` or `navlab.real.sensors` |
| Real SLAM launch/runtime wrappers | `navlab.real.slam` |
| Gazebo truth, SITL-only pose, simulated sensors | `navlab.sim.*` |
| Task registry, Typer CLI, prepare/doctor chains, process backend, artifacts | `orchestration` |

`navlab.common` must stay boring. If a module has ArduPilot arming semantics,
hardware ownership, serial-device assumptions, Gazebo assumptions, or SITL
assumptions, it does not belong in outer `navlab.common`.

## 5. Resolved Boundary State

The active package boundary after this migration is:

| Runtime role | Current owner |
| --- | --- |
| real motor-debug arm/hold/disarm | `navlab.real.companion.nodes.motor_debug` |
| real MAVLink bridge process launcher | `navlab.real.companion.nodes.mavlink_bridge` |
| real FCU controller runtime | transitional: generated by `orchestration.src.tasks.helpers.fcu`; target `navlab.real.companion.nodes.fcu_controller` |
| real FCU telemetry / pose mirror | `navlab.real.companion.nodes.pose_mirror` |
| real MAVLink ExternalNav sender | `navlab.real.companion.nodes.external_nav` |
| real MAVLink IMU bridge | `navlab.real.companion.nodes.imu_bridge` |
| real lidar ROS scan bridge | `navlab.real.companion.nodes.ydlidar_x2_scan` |
| real rangefinder-to-FCU bridge | `navlab.real.companion.nodes.rangefinder_bridge` |
| real ArduPilot FCU/EKF status helpers | `navlab.real.common.fcu_status` |
| Gazebo truth odometry | `navlab.sim.companion.nodes.gazebo_truth_odom` |
| Gazebo truth trajectory recorder | `navlab.sim.companion.nodes.gazebo_truth_trajectory` |
| SITL hover/obstacle missions | `navlab.sim.companion.nodes.hover_mission`, `navlab.sim.companion.nodes.obstacle_mission` |
| SITL/replay odom relay and markers | `navlab.sim.companion.nodes.odom_relay`, `navlab.sim.companion.nodes.vehicle_markers`, `navlab.sim.companion.nodes.world_markers` |
| current scan feature ROS entrypoint | `navlab.sim.companion.nodes.scan_features` until a real entrypoint is needed |

`orchestration` keeps task wrappers, config loading, mode registry, process
lifecycle, and panels. It no longer owns the runtime node implementations listed
above.

Remaining transitional surfaces:

| Package | Status | Target |
| --- | --- | --- |
| `orchestration.src.tasks.helpers.fcu` | still owns generated FCU controller runtime | move direct vehicle-control runtime to `navlab.real.companion.nodes.fcu_controller` |
| `navlab.sim.companion.runtime` | owns current sim companion launcher/config/acceptance | keep here unless a real launcher/config surface becomes necessary |
| `navlab.common.slam` | shared backend/config wrapper used by both real and sim | keep in common while the contract is identical; split only if real/sim behavior diverges |

## 6. Orchestration Contract

`orchestration` may:

- choose runtime mode: real or sim
- read config and task config
- start and stop processes
- run preflight, prepare, common doctor, task doctor
- print panels and write summaries
- pass CLI parameters and env vars into runtime entrypoints

`orchestration` must not:

- own a ROS node implementation
- own real serial sensor parsing
- directly implement vehicle behavior such as arm/hold/disarm, takeoff, landing,
  or guided movement
- mix real and sim runtime assumptions inside one task body

The task wrapper can still call a runtime:

```text
orchestration task
  -> validate config and safety confirmations
  -> start navlab.real.* or navlab.sim.* runtime command
  -> collect return code, logs, summary
```

## 7. Runtime Artifact Contract

Runtime tasks and orchestration communicate through artifact files.

The ownership rule is:

- `orchestration` chooses artifact directories and passes concrete output paths
  to runtime processes.
- `navlab.real.*` or `navlab.sim.*` runtime code writes task result artifacts to
  those paths.
- `orchestration` reads the result artifact, adds orchestration-owned metadata
  such as process logs when needed, and prints panels / returns CLI codes.

The language-neutral schema is:

```text
contracts/proto/navlab/runtime/v1/task_result.proto
```

Current tasks may still write JSON summaries, but migrated summaries should:

- include `schema_version = "navlab.runtime.task_result.v1"`
- stay mappable to `navlab.runtime.v1.TaskResult`
- keep task-specific payloads under structured fields such as `evidence`,
  `metrics`, `details`, or `task_config`
- avoid requiring `orchestration` to import runtime Python modules just to parse
  a result

This lets a future orchestration implementation in Rust or another language read
the same result files by compiling the contract package instead of depending on
Python runtime internals.

`motor-debug` is the current first cut:

```text
orchestration
  -> passes --summary-path artifacts/.../summary.json
navlab.real.companion.nodes.motor_debug
  -> writes summary.json with schema_version
orchestration
  -> reads summary.json and prints the panel
```

## 8. Migration Plan

Move in small cuts.

1. Add target packages:

```text
navlab/real/
navlab/real/common/
navlab/real/companion/nodes/
navlab/sim/
navlab/sim/common/
navlab/sim/companion/nodes/
```

2. Move pure real FCU helpers:

```text
old top-level common FCU status helper
  -> navlab.real.common.fcu_status
```

Update imports. If tests show a truly shared helper, extract only that helper
back into `navlab.common`.

3. Split `motor-debug`:

```text
runtime:
  navlab.real.companion.nodes.motor_debug

wrapper:
  orchestration.src.tasks.built_in.motor_debug
```

The runtime owns MAVLink arm/hold/disarm. The wrapper owns task config, safety
confirmation, artifacts, and process invocation.

4. Move real bridge nodes:

```text
old orchestration lidar bridge runtime
  -> navlab.real.companion.nodes.ydlidar_x2_scan

old orchestration rangefinder bridge runtime
  -> navlab.real.companion.nodes.rangefinder_bridge
```

Then update `orchestration/configs/real_prepare.toml` to launch
`python -m navlab.real.companion.nodes...`.

5. Move sim-only nodes:

```text
old companion Gazebo truth odom runtime
  -> navlab.sim.companion.nodes.gazebo_truth_odom
```

Repeat for simulation-only Gazebo/SITL helpers.

6. Decide shared runtime nodes last.

Move FCU/MAVLink companion nodes to `navlab.real`. If both domains later need
the same core logic, split:

```text
navlab/common/pure_codec_or_math.py
navlab/real/companion/nodes/external_nav.py
navlab/sim/companion/nodes/external_nav.py
```

## 9. Acceptance Checks

Each migration cut should satisfy:

- Import check for moved modules.
- Targeted unit tests for the moved module.
- One orchestration CLI dry path still resolves the new command.
- Real hardware commands are not run as part of migration tests unless
  explicitly requested.

For command migrations, verify the configured command changes from:

```text
python orchestration/src/tasks/.../node.py
```

to:

```text
python -m navlab.real...
```

or:

```text
python -m navlab.sim...
```

## 11. Non-Goals

- Do not rewrite task behavior during package migration.
- Do not change ArduPilot parameters as part of directory movement.
- Do not merge real and sim behavior to reduce duplication if it hides safety
  boundaries.
- Do not make `navlab.common` a new catch-all package.
- Do not require protobuf code generation before a repository-wide codegen
  decision exists.
