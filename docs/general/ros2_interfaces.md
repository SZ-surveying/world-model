# 通用 ROS2 接口设计

## 1. 文档定位

这份文档描述的是**跨场景共用**的 ROS2 接口设计原则。

重点不是规定某一个场景必须有哪些具体 topic，而是定义一套稳定的分层和接口边界，让不同场景可以在同一套系统里复用核心能力。

如果某些接口只适用于特定场景，例如：

- 室内 SLAM 位姿输入
- 特定飞控桥接
- 特定传感器驱动

这些内容应下沉到场景文档或场景适配层，而不是直接写死在通用接口文档里。

## 2. 设计目标

ROS2 侧接口设计建议满足以下目标：

- 时间戳统一
- 坐标系边界清晰
- 自机状态、环境状态、预测结果、控制命令分层发布
- 世界模型节点只做推理，不承担飞控仲裁
- 任何能改变飞行器行为的命令都必须经过安全层
- 多场景之间尽量复用相同的中间表示

## 3. 通用节点分层

建议 ROS2 运行时节点按职责拆成以下几类：

1. 传感器接入节点
2. 时间同步与 TF 节点
3. 状态估计与状态构建节点
4. 世界模型推理节点
5. 任务与规划节点
6. 安全裁决节点
7. 飞控桥接节点

### 3.1 传感器接入节点

职责：

- 接入雷达、相机、IMU、GPS、测距仪等
- 发布原始或轻度标准化数据
- 上报设备健康状态

不应承担：

- 大量业务前处理
- 任务逻辑

### 3.2 时间同步与 TF 节点

职责：

- 对齐设备时间、ROS 时间、飞控时间
- 维护 `map`、`odom`、`base_link`、传感器坐标系之间的关系
- 检测时间回退、跳变和漂移

### 3.3 状态估计与状态构建节点

职责：

- 自机状态整理
- 局部地图构建
- 动态目标跟踪
- 观测融合

这一层的目标是把原始观测转成统一的中间表示。

### 3.4 世界模型推理节点

职责：

- 汇聚过去若干帧状态
- 做未来预测、风险评估或轨迹评分
- 发布结构化模型输出

不应承担：

- 飞控协议处理
- 安全裁决
- 直接控制飞行器

### 3.5 任务与规划节点

职责：

- 接收任务目标
- 结合世界模型和当前状态生成局部目标、轨迹或 setpoint

### 3.6 安全裁决节点

职责：

- 校验所有外发控制命令
- 执行限幅、超时保护、风险检查和降级

### 3.7 飞控桥接节点

职责：

- 把安全层通过的命令转换成飞控接口
- 上报飞控 ACK、模式、健康状态

## 4. 通用 Topic 分层建议

下面不是强制命名，而是推荐按语义分层。

### 观测层

建议主题类别：

- 原始传感器数据
- 预处理结果
- 设备健康状态

示例：

- `/sensors/...`
- `/perception/raw/...`
- `/diagnostics/...`

### 自机状态层

建议主题类别：

- 自机位姿
- 自机速度
- 飞控模式和健康状态
- 状态估计质量

示例：

- `/ego/state`
- `/flight/status`
- `/flight/health`

### 环境状态层

建议主题类别：

- 局部占据图
- 障碍物跟踪结果
- 自由空间
- 地图调试信息

示例：

- `/world/occupancy`
- `/world/tracks`
- `/world/free_space`

### 世界模型输出层

建议主题类别：

- 未来占据预测
- 风险图
- 候选轨迹评分
- 模型健康状态

示例：

- `/world/predicted_occupancy`
- `/world/risk_map`
- `/world/model_status`

### 规划层

建议主题类别：

- 任务目标
- 局部轨迹
- 速度指令
- 规划调试信息

示例：

- `/mission/goal`
- `/planner/trajectory`
- `/planner/cmd_vel`

### 安全与执行层

建议主题类别：

- 最终飞行命令
- 安全事件
- 安全状态
- 飞控 ACK

示例：

- `/flight/command`
- `/safety/event`
- `/safety/state`
- `/flight/ack`

## 5. 通用消息设计原则

建议把系统核心消息放在独立接口包里，例如：

- `wm_interfaces`

消息设计时建议遵循以下原则：

- 字段语义稳定
- 时间戳明确
- 坐标系明确
- 状态与命令分开
- 健康状态与业务数据分开
- 能支持录包回放和跨场景复用

## 6. 推荐的核心消息类型

通用层建议至少抽象这些消息：

- `EgoState`
- `FlightStatus`
- `FlightHealth`
- `TrackArray`
- `Occupancy` 或等价局部地图表示
- `RiskMap`
- `TrajectorySetpoint`
- `FlightCommand`
- `SafetyEvent`
- `ModelStatus`

具体字段可以在实现阶段进一步收敛，但消息语义边界应尽早稳定。

## 7. 坐标系与时间的通用要求

这是接口设计中最容易出问题的部分之一。

建议所有场景都明确：

- 每个 topic 属于哪个参考系
- 每条消息的时间戳来自哪里
- 是否存在延迟补偿
- 状态估计坐标系和飞控坐标系如何映射

如果某个场景存在特殊转换，例如 ENU / NED、FLU / FRD，应在场景文档中单独写清楚，但通用接口层必须预留这些边界。

## 8. 多场景复用原则

为了支持多个场景，ROS2 接口不应直接把场景假设写死进消息层。

例如不建议：

- 在通用消息里强行绑定某一种传感器
- 在通用 topic 命名里写死某一种飞控
- 在通用节点职责里混入某个场景特有逻辑

建议做法是：

- 通用 topic 保持语义中立
- 场景差异收敛在 adapter、config 和 launch 层
- 通用中间表示尽量一致

## 9. 接口结论

对这个仓库来说，ROS2 接口层最重要的不是“先把所有 topic 命名定死”，而是先把边界定清楚：

- 谁负责原始观测
- 谁负责状态构建
- 谁负责模型输出
- 谁负责最终执行

只要这层边界清晰，多个场景就能在一套相对稳定的接口体系上演进。
