# Stage 1 MAVLink ODOMETRY Mapping

## Decision

Stage 1 currently keeps `/ap/tf` as a ROS-side diagnostic and future DDS entry point, but the proven SITL injection path is MAVLink v2 `ODOMETRY`.

Current main path:

```text
/odom
  -> external_nav_bridge
  -> /external_nav/odom
  -> navlab.companion.nodes.external_nav
  -> MAVLink ODOMETRY
  -> ArduPilot SITL tcp:sitl:5762
```

Parallel diagnostic path:

```text
/odom
  -> external_nav_bridge
  -> /ap/tf
```

`/ap/tf` is still recorded because it proves the bridge can produce the ArduPilot DDS-shaped transform, but Stage 1 does not rely on DDS for the accepted SITL closed loop.

## ROS Entry Point

MAVLink `ODOMETRY` output is generated by:

```text
navlab.companion.nodes.external_nav
```

Default CLI contract:

```bash
python3 -m navlab.companion.nodes.external_nav \
  --endpoint tcp:sitl:5762 \
  --odom-topic /external_nav/odom \
  --status-topic /mavlink_external_nav/status \
  --rate-hz 20 \
  --quality 100 \
  --reset-counter 0 \
  --source-system 191
```

Status topic:

```text
/mavlink_external_nav/status
```

The status message is JSON and includes endpoint, sent count, rate, input age, MAVLink frames, quality, reset counter, estimator type, time source, and the field mapping used by the sender.

## Field Mapping

Input semantics:

```text
nav_msgs/msg/Odometry on /external_nav/odom
ROS position: ENU-like local frame
ROS body twist: FLU-like body frame
```

Output semantics:

```text
MAVLink message: ODOMETRY
frame_id: MAV_FRAME_LOCAL_FRD
child_frame_id: MAV_FRAME_BODY_FRD
estimator_type: MAV_ESTIMATOR_TYPE_VIO
component: MAV_COMP_ID_VISUAL_INERTIAL_ODOMETRY
```

Mapping:

| MAVLink field | Source |
| --- | --- |
| `time_usec` | sender monotonic clock in microseconds |
| `frame_id` | fixed `MAV_FRAME_LOCAL_FRD` |
| `child_frame_id` | fixed `MAV_FRAME_BODY_FRD` |
| `x` | `odom.pose.pose.position.x` |
| `y` | `-odom.pose.pose.position.y` |
| `z` | `-odom.pose.pose.position.z` |
| `q` | `[w, x, -y, -z]` from `odom.pose.pose.orientation` |
| `vx` | `odom.twist.twist.linear.x` |
| `vy` | `-odom.twist.twist.linear.y` |
| `vz` | `-odom.twist.twist.linear.z` |
| `rollspeed` | `odom.twist.twist.angular.x` |
| `pitchspeed` | `-odom.twist.twist.angular.y` |
| `yawspeed` | `-odom.twist.twist.angular.z` |
| `pose_covariance` | upper-triangular 6x6 from `odom.pose.covariance` |
| `velocity_covariance` | upper-triangular 6x6 from `odom.twist.covariance` |
| `reset_counter` | CLI `--reset-counter`, default `0` |
| `estimator_type` | fixed `MAV_ESTIMATOR_TYPE_VIO` |
| `quality` | CLI `--quality`, default `100` |

## Reset Counter

Stage 1 uses `reset_counter=0` because the synthetic and TF-backed odometry used in this stage does not publish reset events.

Future SLAM integration must increment `reset_counter` when the localization origin or pose estimate has a discontinuous reset. If reset events are added as ROS messages later, the sender should translate them into this field instead of keeping the static CLI value.

## Rate

Stage 1 sends MAVLink `ODOMETRY` at `20 Hz`.

This rate is independent from the bridge status publication rate. The sender reuses the latest `/external_nav/odom` sample on every send tick and exposes the input age in `/mavlink_external_nav/status`.

## Boundaries

- `external_nav_bridge` remains the internal ROS normalization point: `/odom -> /external_nav/odom + /external_nav/status + /ap/tf`.
- `mavlink_external_nav_sender` is the only component allowed to translate `/external_nav/odom` into MAVLink `ODOMETRY`.
- `/ap/tf` and MAVLink `ODOMETRY` must not both be treated as independent control sources for the same ArduPilot run.
- Stage 1 accepted SITL runs use MAVLink injection on `tcp:sitl:5762`; DDS remains a future or alternate path.
